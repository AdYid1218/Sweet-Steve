 # 实验三：ZYNQ串口接收与HDMI显示扩展
 
 ## 一、实验目的
 
 本实验以 Xilinx ZYNQ-7020（XC7Z020）为核心平台，构建一条 **PC → UART → PS → AXI BRAM → PL → HDMI** 的完整图像传输与显示链路，在此基础上完成以下四项目标：
 
 1. **掌握 ZYNQ PS-PL 协同设计流程**：在 Vivado 中搭建包含 ZYNQ7 Processing System、AXI BRAM Controller 和 Block Memory Generator 的 Block Design，并在 SDK 中编写 PS 端 C 程序驱动 UART 外设接收数据并写入 BRAM。
 2. **理解 HDMI 显示时序与 BRAM 读取逻辑**：PL 端通过 `hdmi_bram_display.v` 产生 720p 时序，从 BRAM 端口 B 读取像素数据，经 RGB2DVI 编码输出 HDMI 信号。
 3. **实现串口通信协议解析**：PS 端程序按照帧头（`0x55 0xAA`）+ 行头（`0x33 0xCC`）+ 像素数据的协议，完整接收来自 PC 端 `camera_uart_sender.py` 的 RGB888 图像帧。
 4. **完成四项可选扩展**：修改 HDMI 背景颜色为天蓝色、将 128×72 图像 1:1 居中显示于 1280×720 屏幕、为图像添加红色边框、以及通过调整 PC 端发送参数记录串口帧率与稳定性数据。

 ## 二、实验环境与系统架构
 
 ### 2.1 硬件平台
 
 | 组件 | 型号 / 参数 |
 | --- | --- |
 | 开发板 | ZYNQ-7020（XC7Z020CLG484-1） |
 | 处理器 | ARM Cortex-A9 双核（PS 端） |
 | FPGA 逻辑 | Artix-7 等效架构（PL 端） |
 | 串口芯片 | USB-UART（FTDI / CP210x），115200 baud |
 | 显示器 | ThinkVision HDMI 显示器，支持 720p |
 | PC 端 | Windows，Python 3.13（Conda 虚拟环境 `fpga`） |
 
 ### 2.2 系统数据流架构
 
 ```
 PC 摄像头/图片                    ZYNQ-7020                     HDMI 显示器
 ┌──────────────┐    USB-UART    ┌──────────────────────────────────┐   ┌─────────┐
 │  camera_uart  │ ──────────→  │  PS UART → 帧解析 → AXIGP0      │   │ Think-  │
 │  _sender.py   │ 115200 8N1   │      → BRAM Controller → BRAM   │   │ Vision  │
 │  (Python)     │              │                    ↓ (PORT B)    │   │  720p   │
 └──────────────┘              │          PL hdmi_bram_display    │──→│         │
                               │              → RGB2DVI           │   └─────────┘
                               └──────────────────────────────────┘
 ```
 
 - **PC 端**：`camera_uart_sender.py` 通过串口发送自定义协议帧（帧头 + 行头 + RGB888 像素），可选用摄像头实时画面或本地图片。
 - **PS 端**：`main.c` 在 ARM Cortex-A9 上轮询 UART 外设，解析协议后将逐行像素以 `0x00RRGGBB` 格式写入 AXI BRAM（基地址 `0x40000000`，范围 64 KB）。
 - **PL 端**：`hdmi_bram_display.v` 生成 720p 时序（74.25 MHz 像素时钟），从 BRAM 端口 B 同步读取图像数据，经过两拍延迟对齐后进行像素输出。
 - **HDMI 输出**：经 `rgb2dvi` IP 核将并行 RGB 与同步信号编码为 TMDS 差分对，驱动 HDMI 显示器。
 
 ### 2.3 图像参数与地址映射
 
 | 参数 | 值 |
 | --- | --- |
 | 原始图像分辨率 | 128 × 72 |
 | HDMI 输出分辨率 | 1280 × 720 (720p) |
 | 显示模式 | **1:1 居中显示**（扩展修改后） |
 | 像素格式 | RGB888，每像素 32 bit 存储（高 8 位保留） |
 | BRAM 地址计算 | `0x40000000 + ((y × 128 + x) << 2)` |

 ## 三、硬件代码设计与修改解析
 
 ### 3.1 顶层模块结构
 
 硬件层次自顶向下为：
 
 ```
 top.v
  └── hdmi_pl_top.v
       ├── hdmi_bram_display.v    ← 核心显示逻辑（本实验修改对象）
       └── rgb2dvi_0              ← HDMI 编码 IP
 ```
 
 ### 3.2 原始代码的放大显示逻辑
 
 在未修改前，`hdmi_bram_display.v` 通过两个缩放常数将 128×72 的图像放大 10 倍：
 
 ```verilog
 localparam SCALE_X = H_ACTIVE / IMG_WIDTH;   // 1280 / 128 = 10
 localparam SCALE_Y = V_ACTIVE / IMG_HEIGHT;  // 720  / 72  = 10
 ```
 
 坐标映射采用整除方式，使得扫描到任意 `active_x` / `active_y` 时，均通过 `active_x / SCALE_X` 和 `active_y / SCALE_Y` 映射到 BRAM 读地址。这一策略虽然实现了全屏填充，但导致图像被粗暴拉伸，出现严重的马赛克颗粒感，且无法支持背景颜色定制或边框叠加。
 
 ### 3.3 四项扩展的代码修改详解
 
 本实验一次性完成了 README 第 13 节所列的全部四项扩展，其中前三项通过修改 `hdmi_bram_display.v` 实现。下面逐一解析每个修改的技术细节。
 
 ---
 
 #### **扩展 1：修改 HDMI 背景颜色（天蓝色）**
 
 **设计思路**：将非图像区域的 RGB 输出从全黑（`24'h000000`）替换为天蓝色（`24'h87CEEB`）。核心在于将 RGB 输出从单一的"图像/消隐"二值选择改为三路选择器：**图像区域 → 边框区域 → 背景区域 → 消隐区**。
 
 **关键代码段**（RGB 输出组合逻辑）：
 
 ```verilog
 assign rgb_r = de_reg_d1 ? (image_area_d1 ? pixel_reg[23:16] :
                 (border_area_d1 ? 8'hFF : 8'h87)) : 8'h00;
 assign rgb_g = de_reg_d1 ? (image_area_d1 ? pixel_reg[15:8]  :
                 (border_area_d1 ? 8'h00 : 8'hCE)) : 8'h00;
 assign rgb_b = de_reg_d1 ? (image_area_d1 ? pixel_reg[7:0]   :
                 (border_area_d1 ? 8'h00 : 8'hEB)) : 8'h00;
 ```
 
 - `de_reg_d1` 为打两拍后的数据使能信号，控制消隐期强制输出全黑（`8'h00`）。
 - 若处于非消隐状态且非图像区、非边框区，则三个通道分别输出 `8'h87`、`8'hCE`、`8'hEB`，组合为天蓝色 `#87CEEB`。
 - 这一修改仅涉及组合逻辑层面的色彩选择，不增加时序路径深度，对时序闭合无负面影响。
 
 ---
 
 #### **扩展 2：调整图像显示位置（1:1 居中显示）**
 
 **设计思路**：取消原始代码中的 10 倍缩放常数 `SCALE_X` 和 `SCALE_Y`，改用静态起始坐标 `IMG_X_START` 和 `IMG_Y_START`，将 128×72 图像定位在 1280×720 有效区域的几何中心。
 
 **坐标计算**：
 
 ```verilog
 localparam IMG_X_START = (H_ACTIVE - IMG_WIDTH)  / 2;  // (1280-128)/2 = 576
 localparam IMG_Y_START = (V_ACTIVE - IMG_HEIGHT) / 2;  // (720-72)/2   = 324
 ```
 
 **区域判定线网**：
 
 ```verilog
 assign image_area = (active_x >= IMG_X_START) &&
                     (active_x <  IMG_X_START + IMG_WIDTH) &&
                     (active_y >= IMG_Y_START) &&
                     (active_y <  IMG_Y_START + IMG_HEIGHT);
 ```
 
 **局部坐标映射**：仅在 `image_area` 为真时，计算像素在图像内部的相对坐标，生成 BRAM 读地址：
 
 ```verilog
 assign image_x = image_area ? (active_x - IMG_X_START) : 7'd0;
 assign image_y = image_area ? (active_y - IMG_Y_START) : 7'd0;
 assign image_word_addr = {image_y, 7'b0} + {7'd0, image_x};
 ```
 
 - 通过组合逻辑减法得到相对坐标，不引入额外时钟周期。
 - BRAM 地址生成公式 `{image_word_addr, 2'b00}` 对应 BRAM 按 32-bit 字编址（每像素 4 字节）。
 - 此修改将图像从全屏 10 倍拉伸恢复为 1:1 原始分辨率，同时位于屏幕正中央，画面质量大幅提升。
 
 ---
 
 #### **扩展 3：增加图像红色边框**
 
 **设计思路**：在 `image_area` 外侧扩展 `BORDER_WIDTH`（4 像素）宽的环形区域，定义为 `border_area`，在该区域强制输出红色（`24'hFF0000`）。
 
 **边框区域判定线网**：
 
 ```verilog
 localparam BORDER_WIDTH = 12'd4;
 
 assign border_area = (active_x >= (IMG_X_START - BORDER_WIDTH)) &&
                      (active_x <  (IMG_X_START + IMG_WIDTH + BORDER_WIDTH)) &&
                      (active_y >= (IMG_Y_START - BORDER_WIDTH)) &&
                      (active_y <  (IMG_Y_START + IMG_HEIGHT + BORDER_WIDTH)) &&
                      (!image_area);
 ```
 
 - 外层矩形范围相比图像区域在各方向外扩 `BORDER_WIDTH` 像素。
 - 通过 `(!image_area)` 排除内部图像区域，确保 `border_area` 仅覆盖边框本身的像素，不覆盖或破坏图像内容。
 - 边框输出优先级高于背景但低于图像，在 RGB 三路选择器中已体现。
 
 ---
 
 #### **流水线延迟对齐策略（BRAM 读取延迟补偿）**
 
 BRAM 从地址给出到数据输出需两个时钟周期。为补偿此延迟，代码引入了两条延迟链：
 
 ```verilog
 // 同步信号打两拍 (d0 → d1)
 hs_reg_d0 <= hs_reg;    vs_reg_d0 <= vs_reg;    de_reg_d0 <= de_reg;
 hs_reg_d1 <= hs_reg_d0; vs_reg_d1 <= vs_reg_d0; de_reg_d1 <= de_reg_d0;
 
 // 区域判定信号同步打两拍
 image_area_d0 <= image_area;    border_area_d0 <= border_area;
 image_area_d1 <= image_area_d0; border_area_d1 <= border_area_d0;
 
 // BRAM 数据直接寄存器锁存
 pixel_reg <= bram_dout[23:0];
 ```
 
 - `bram_addr_reg` 和 `bram_en` 在 `clk` 上升沿发出。
 - BRAM 在下一个上升沿输出 `bram_dout`，`pixel_reg` 在同一周期锁存。
 - `image_area_d1` / `border_area_d1` 同步至数据到达时刻，用于驱动 RGB 组合逻辑的选路。
 - 经过这一流水线设计，`hs_reg_d1` / `vs_reg_d1` / `de_reg_d1` 与 `pixel_reg` 严格对齐，消除了显示错位风险。

 ## 四、软件与外设联调
 
 ### 4.1 PS 端软件逻辑
 
 SDK 工程 `ps_uart_bram_app` 中的 `main.c` 实现以下流程：
 
 ```
 初始化 UART（115200, 8N1）
 初始化 BRAM 地址指针 → 0x40000000
 打印 "PS UART BRAM HDMI display"
 打印 "BRAM base: 0x40000000, baud: 115200"
 循环：
     等待 0x55 0xAA 帧头
     若匹配 → 解析 width / height / format
     若不符合预期 → 打印 "frame header error -1" 重新等待
     遍历每行：
         等待 0x33 0xCC 行头
         解析行号
         读取 128 × 3 字节像素数据
         逐像素写入 BRAM（0x00RRGGBB）
     行尾更新帧计数器
     打印 "frame received: xxx"
 ```
 
 串口参数配置严格匹配 PC 端：
 
 | 参数 | 值 |
 | --- | --- |
 | 波特率 | 115200 |
 | 数据位 | 8 |
 | 停止位 | 1 |
 | 校验位 | None |
 | 流控 | None |
 
 ### 4.2 PC 端 Python 发送脚本
 
 `camera_uart_sender.py` 支持两种输入模式和灵活的参数调节：
 
 **摄像头实时模式**：
 
 ```bash
 python camera_uart_sender.py --port COM7 --baud 115200 --camera 0 --fps 0.2 --preview
 ```
 
 **图片发送模式**：
 
 ```bash

 ```
 
 关键参数说明：`--fps` 控制帧率上限，`--line-delay` 控制行间额外延时（单位秒）。这两个参数是扩展 4 的调优对象。

 ## 五、实验结果验证与分析
 
 ### 5.1 硬件综合与实现验证
 
 #### 时序分析
 
 ![硬件时序验证](./实验三时序截图.png)
 
 **图文解析**：从 Timing Summary 报告可知，系统的 **Worst Negative Slack (WNS) 为 8.117 ns**，**Total Negative Slack (TNS) 为 0 ns**，且报告中明确标注 **"All user specified timing constraints are met"**。这一结果充分证明了以下两点：
 
 - **时序闭合**：PL 端在 74.25 MHz 像素时钟（720p 标准时钟）下，所有数据路径均满足建立时间要求，不存在时序违例。
 - **流水线设计安全**：经过两拍延迟对齐后的 BRAM 读取逻辑、HS/VS/DE 同步信号以及 RGB 输出组合逻辑均被正确综合，未产生关键路径瓶颈。
 
 8.117 ns 的建立时间余量意味着当前设计具有充分的时序裕度，即使在更高的时钟频率或更复杂的逻辑层级下仍能稳定运行。
 
 #### 资源利用率分析
 
 ![资源利用率](./实验三资源利用.png)
 
 **图文解析**：从 Utilization 报告可知：
 
 | 资源类型 | 使用量 | 可用总量 | 利用率 |
 | --- | --- | --- | --- |
 | Slice LUTs | 约 4% | 53,200 | 极低 |
 | Slice Registers | 约 2% | 106,400 | 极低 |
 | Block RAM Tile | 16 | 140 | 11.4% |
 | DSP48E1 | 0 | 220 | 0% |
 | IO / BUFG | 少量 | — | 正常 |
 
 - **逻辑资源消耗极低**：LUT 和寄存器用量在 4% 以下，表明 HDMI 时序生成和 BRAM 读取逻辑非常轻量，整个设计以数据通路和状态控制为主，不含乘加运算等组合密集型电路。
 - **BRAM 占用合理**：16 个 Block RAM Tile 对应 AXI BRAM Controller 例化的 64 KB 存储空间，用于缓存一帧 128×72 的 RGB888 图像（实际约 27 KB），完全满足一帧数据的存储需求。
 - **DSP 资源零占用**：不涉及图像滤波或神经网络计算，DSP48E1 无需使用。

 ### 5.4 串口帧率与稳定性测试（扩展 4）
 
 本扩展不涉及硬件代码修改，仅调整 PC 端 Python 脚本的运行参数，记录不同帧率和行延时条件下的传输稳定性。测试条件如下：
 
 | 硬件环境 | 参数 |
 | --- | --- |
 | 串口波特率 | 115200 baud |
 | BRAM 写入方式 | 单帧逐行写入 |
 | PC 端脚本 | `camera_uart_sender.py` |
 
 #### 理论带宽分析
 
 在 115200 baud 的 UART 8N1 模式下，实际有效字节传输率约为：
 
 ```
 115200 / 10 = 11520 byte/s
 ```
 
 一帧 128×72 的 RGB888 图像数据量为：
 
 ```
 128 × 72 × 3 = 27648 byte（像素）
 + 帧头 6 byte + 72 × 行头 4 byte ≈ 27942 byte
 ```
 
 理论最高帧率约为 `11520 / 27942 ≈ 0.41 fps`。因此 `--fps 0.2`（5 秒/帧）可稳定通信，而 `--fps 0.5` 或更高则可能发生缓冲区溢出。
 
 #### 测试记录
 
 | 测试组 | `--fps` | `--line-delay` | 画面输出 | 串口报错 | 分析 |
 | --- | --- | --- | --- | --- | --- |
 | 1 | 0.2 | 0（默认） | 完整刷新，居中+边框+背景均正常 | 无 | 稳定区间，数据速率远低于理论上限 |
 | 2 | 0.33 | 0（默认） | 画面可刷新，偶尔出现半帧 | 偶现 `frame error -2`（行号不匹配） | 接近理论上限，发送缓冲不够稳定 |
 | 3 | 0.33 | 0.001 | 画面基本完整，无明显错行 | 偶现 `frame error -5`（等待行头超时） | `line-delay` 缓解了 PS 端处理压力 |
 | 4 | 0.5 | 0（默认） | 画面频繁中断 | 频繁 `frame error -5 / -6 / -7` | 超出 115200 带宽上限，不可用 |
 | 5 | 0.5 | 0.002 | 仍出现明显丢行 | 频繁 `frame error` | 单纯增加行延时无法弥补帧率超限 |
 | 6 | 1.0 | 0（默认） | 几乎无法刷新 | 大量连续 `frame error` | 串口缓冲区完全溢出 |
 
 #### 测试结论
 
 - **推荐工作参数**：`--fps 0.2`（无需 `--line-delay`），在该参数下系统运行稳定，连续观察超过 50 帧未出现任何 `frame error`。
 - **稳定性边界**：`--fps 0.33` 配合 `--line-delay 0.001` 可在大多数情况下正常运行，但偶有行超时，属于临界状态。
 - **不可用区间**：`--fps ≥ 0.5` 时传输可靠性显著下降，串口缓冲区持续溢出。这是由 115200 baud 的物理带宽上限决定的。若要提高帧率，建议后续升级至更高波特率（如 921600）或改用 USB/Ethernet/DMA 等更高带宽的数据通道。

 ## 六、实验总结与拓展探讨
 
 ### 6.1 实验成果总结
 
 本实验在 ZYNQ-7020 平台上构建了完整的 **PC 端 UART → PS 端协议解析 → AXI BRAM 缓存 → PL 端 HDMI 显示** 的图像传输与显示链路，并在此基础上完成了全部四项可选扩展：
 
 | 扩展 | 完成情况 | 核心实现 |
 | --- | :-: | --- |
 | 修改 HDMI 背景颜色 | ✅ | RGB 三路选择器，非图像区输出天蓝色 `#87CEEB` |
 | 调整图像显示位置 | ✅ | 取消 10 倍缩放，居中坐标 `(576, 324)` 实现 1:1 显示 |
 | 增加图像边框 | ✅ | `border_area` 判定线网 + 红色输出，4 像素宽 |
 | 串口帧率与稳定性记录 | ✅ | 6 组 `--fps` / `--line-delay` 对比测试，记录报错边界 |
 
 ### 6.2 技术要点复盘
 
 1. **坐标映射重构**：从全屏 10 倍拉伸改为 1:1 居中显示，核心在于将全局除法的缩放映射替换为局部坐标减法 + 区域判定组合逻辑，这一思路可推广至任意分辨率图像的灵活定位。
 2. **流水线对齐设计**：BRAM 读取的两拍延迟要求在同步信号和区域判定信号上同时补偿延迟，否则会出现 DE 与像素数据错位导致的画面闪烁或撕裂。
 3. **组合逻辑与优先级**：RGB 三路选择器采用 **图像 > 边框 > 背景 > 消隐** 的优先级设计，通过嵌套条件运算符实现清晰的分层渲染，且不引入额外时序开销。
 4. **带宽认知**：串口传输的帧率受限于波特率物理上限，实验数据表明 115200 baud 在 128×72 RGB888 格式下的安全帧率约为 0.2 fps。
 
 ### 6.3 后续拓展方向
 
 - **提高传输带宽**：将 UART 波特率升级至 921600（需 PS 端和 PC 端同步修改），或改用 USB 虚拟以太网 / AXI DMA 实现高速批量传输。
 - **图像处理集成**：在 PL 端接入 Sobel 边缘检测 IP 核，对 BRAM 中的图像数据进行实时滤波后再输出，过渡至 Sobel 系列实验的下一阶段。
 - **双缓冲机制**：在 BRAM 中分配两个帧缓冲（Ping-Pong Buffer），PS 端写入缓冲区 A 时 PL 端读取缓冲区 B，帧切换时交换角色，消除画面撕裂。
 - **OSD 叠加显示**：在边框区域扩展实现 OSD 文本或图标叠加，如实时帧率、时间戳显示。
